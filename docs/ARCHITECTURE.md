# Architecture

## Overview

`bessa_patterns.ts` is a zero-dependency TypeScript library that provides concrete, generic implementations of classic design patterns. Each pattern lives in its own file under `src/` and is exported from a single public barrel (`src/index.ts`).

## Directory Structure

```
bessa_patterns.ts/
├── src/
│   ├── index.ts                  # Public barrel export (all patterns)
│   ├── ObserverSubject.ts        # Observer pattern — typed callback variant
│   └── DualObserverSubject.ts    # Observer pattern — GoF + function-based dual variant
├── test/
│   ├── ObserverSubject.test.ts
│   └── DualObserverSubject.test.ts
├── dist/                         # Compiled output (CommonJS + type declarations)
├── docs/
│   ├── README.md                 # Documentation index and navigation
│   ├── GETTING_STARTED.md
│   ├── ARCHITECTURE.md           # (this file)
│   ├── API.md
│   ├── ROADMAP.md
│   ├── OBSERVER_SUBJECT_API.md
│   └── DUAL_OBSERVER_SUBJECT_API.md
├── scripts/
│   └── deploy.sh                 # Guarded npm publish script (used by ai-workflow deploy)
└── .github/
    ├── copilot-instructions.md   # GitHub Copilot workspace context
    └── workflows/
        └── ci.yml                # GitHub Actions CI (Node 18 & 20)
```

## Directory Guide

| Directory            | Purpose                                                                                                            |
| -------------------- | ------------------------------------------------------------------------------------------------------------------ |
| `src/`               | TypeScript source — one file per pattern, exported from `src/index.ts`                                             |
| `test/`              | Jest test suites — one file per source file, co-located by name                                                    |
| `dist/`              | Compiled CommonJS output + `.d.ts` declarations (git-ignored, generated by `npm run build`)                        |
| `docs/`              | Project documentation — API references, architecture, roadmap, guides. See [`docs/README.md`](README.md) for index |
| `scripts/`           | Automation scripts. `deploy.sh` guards and executes `npm publish`; invoked by `ai-workflow deploy`                 |
| `.github/`           | GitHub-specific configuration: Copilot workspace instructions (`copilot-instructions.md`) and CI/CD workflows      |
| `.github/workflows/` | GitHub Actions workflows. `ci.yml` runs lint → test → build on Node 18 & 20 for every push/PR to `main`            |

## Pattern Catalogue

### ObserverSubject\<T\>

**File:** `src/ObserverSubject.ts`

A generic, concrete Subject that manages typed callback observers. Observers are plain functions `(snapshot: T) => void` — no interface to implement.

```
┌─────────────────────────────┐
│     ObserverSubject<T>      │
├─────────────────────────────┤
│ - _observers: Function[]    │
├─────────────────────────────┤
│ + subscribe(fn) → unsub     │
│ + unsubscribe(fn) → bool    │
│ + getObserverCount() → num  │
│ + clearObservers()          │
│ # _notifyObservers(T)       │
└─────────────────────────────┘
```

Intended for direct use or subclassing. The `_notifyObservers` prefix signals that callers should be the owning class or a subclass, not external consumers.

### DualObserverSubject

**File:** `src/DualObserverSubject.ts`

A concrete Subject that maintains **two independent observer collections** — one for GoF-style objects (`{ update?(...args) }`) and one for plain functions (`(...args) => void`). Notifying one channel has no effect on the other.

```
┌──────────────────────────────────────┐
│           DualObserverSubject        │
├──────────────────────────────────────┤
│ - _observers: ObserverObject[]       │   ← GoF channel
│ - _functionObservers: Function[]     │   ← function channel
├──────────────────────────────────────┤
│ + subscribe(obj) / unsubscribe(obj)  │
│ + subscribeFunction(fn) / unsub...   │
│ + notifyObservers(...args)           │   → calls obj.update()
│ + notifyFunctionObservers(...args)   │   → calls fn()
│ + getObserverCount()                 │
│ + getFunctionObserverCount()         │
│ + clearAllObservers()                │
└──────────────────────────────────────┘
```

## Design Principles

| Principle                     | How it's applied                                                                     |
| ----------------------------- | ------------------------------------------------------------------------------------ |
| **Typed first**               | Generics over `any`; `unknown` where the type cannot be fixed at the class level     |
| **Concrete by default**       | All classes are directly instantiable; no abstract base required                     |
| **Error isolation**           | Per-observer `try/catch` — one bad observer cannot block others                      |
| **Immutable collections**     | `subscribe`/`unsubscribe` create new arrays (spread + filter), never mutate in place |
| **Zero runtime dependencies** | `dependencies` in `package.json` is empty                                            |
| **One file per pattern**      | `src/<PatternName>.ts` + `test/<PatternName>.test.ts`                                |

## Build Pipeline

```
TypeScript source (src/)
        │
        ▼ tsc (CommonJS, ES2020 target)
Compiled output (dist/)
  ├── index.js          ← require('bessa_patterns.ts')
  ├── index.d.ts        ← type declarations
  ├── ObserverSubject.js / .d.ts
  └── DualObserverSubject.js / .d.ts
```

## Test Architecture

Tests use **Jest + ts-jest** (CommonJS mode). Each pattern file has a corresponding test file covering:

- Constructor initialisation
- All public methods (happy path + edge cases)
- Error isolation behaviour
- Immutable array pattern (collection identity checks)

## CI / Deploy

**CI** (`.github/workflows/ci.yml`): on every push/PR to `main`, runs lint → test → build across Node 18 and 20.

**Deploy** (`scripts/deploy.sh`): invoked via `ai-workflow deploy`. Guards: `NPM_TOKEN` present, tests pass, build succeeds — then publishes to npm.
